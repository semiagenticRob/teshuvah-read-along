# Audio-Visual Sync Troubleshooting Protocol

## Ground Truth (NEVER modify)

| Asset | Location | What it is |
|---|---|---|
| **MP3 audio files** | `assets/audio/{service}/{prayer_id}.mp3` | The actual audio the user hears. Generated by ElevenLabs. These are the immutable source of truth. |
| **Whisper raw transcriptions** | `assets/whisper_raw/{service}/{prayer_id}.json` | Word-level timestamps from Whisper medium model run against the MP3s. These represent what a speech recognition model detected in the audio. Format: `[{ "text": "מודה", "start": 0, "end": 440 }, ...]` |
| **Bundled Hebrew text** | `src/data/bundled/{service}/{prayer_id}.json` | The canonical Hebrew text from Sefaria, split into sections. The `he` array contains the text the user sees on screen. This defines the visual word sequence. |

## Derived Data (CAN be regenerated/modified)

| Asset | Location | Generated by | What it is |
|---|---|---|---|
| **Timing JSON** | `assets/timing/{service}/{prayer_id}.json` | `scripts/whisper_align_timing.py` | Word-level start/end timestamps mapped to the bundled text structure. This is what the app loads to sync highlighting with audio. |

## App Code (modify with care)

| File | Role |
|---|---|
| `src/data/timingAssets.ts` | Loads timing JSON files, maps prayer ID → timing data |
| `src/api/prayerTextService.ts` | `applyTimingData()` maps timing sections to visual lines |
| `src/hooks/useWordSync.ts` | Polls audio position, binary-searches word index, sets highlighted word. Contains `ANTICIPATION_OFFSET_MS = 85` |
| `src/components/ReadAlongView.tsx` | Renders highlighted word based on `currentLineIndex` + `currentWordIndex` |
| `src/hooks/useAudioPlayer.ts` | expo-av playback, provides `getPositionMs()` |

## Known Constants

- **Render pipeline latency**: ~50ms (React Native reconciliation + native bridge + frame)
- **Anticipation offset**: 85ms (compensates for render latency)
- **Sync polling interval**: 50ms
- **expo-av getPositionMs latency**: 0-2ms

## Per-Prayer Troubleshooting Procedure

### Step 1: Inspect Alignment Quality

Run for the prayer you're testing:

```bash
python3 scripts/inspect_prayer.py {service} {prayer_id}
```

This script (below) shows:
- Side-by-side Whisper word ↔ bundled word alignment
- Any 50ms sliver words (overlap victims)
- Any gaps > 500ms between consecutive words
- Total timing coverage vs MP3 duration

**What to look for:**
- `MISMATCH` — Whisper word doesn't match bundled word (potential misalignment)
- `SLIVER` — Word has ≤ 50ms duration (will be skipped visually)
- `GAP` — Large gap between words (highlight will freeze)

### Step 2: On-Device Playback Test

Play the prayer in the app. Watch for:
- Words that are never highlighted (slivers)
- Highlight jumping backward
- Highlight freezing for more than ~1 second
- Audio ahead of highlight (systematic drift)

Note the specific Hebrew word where issues occur.

### Step 3: Cross-Reference

Find the problem word in the Step 1 output. Common causes:

| Symptom | Cause | Fix |
|---|---|---|
| Word skipped | 50ms sliver from overlap | Alignment merging too many Whisper words |
| Highlight jumps backward | Non-monotonic timestamps | Alignment ordering issue |
| Highlight freezes 2+ seconds | Gap in timing data | Whisper missed that section of audio |
| Audio ahead by consistent amount | Timing data systematically late | May need prayer-specific offset |
| Audio ahead and getting worse | Timestamps diverge from audio | Alignment mapped wrong Whisper words |

### Step 4: Fix (if needed)

If the alignment is wrong for specific words, the fix is in `whisper_align_timing.py`. Re-run:

```bash
python3 scripts/whisper_align_timing.py
```

This regenerates ALL timing JSONs from the Whisper raw data. It's safe to re-run — it reads from `whisper_raw/` (ground truth) and writes to `assets/timing/` (derived).

If a specific prayer needs manual timestamp adjustment, edit the timing JSON directly. But prefer fixing the alignment script so the fix applies to all prayers.

### Step 5: Verify Fix

Re-run Step 1 to confirm the alignment improved, then repeat Step 2.
